## Critical Bug Fix

### Compression Floor Guard — Full Payload Check (fix)

**Severity:** High — causes session termination on otherwise functional conversations  
**Introduced in:** v0.1.5 (commit `15f37ac2e`, upstream-sync 2026-06-14)  
**Fixed in:** commit `54a5cd469`

#### The Problem

The `MINIMUM_CONTEXT_LENGTH` guard (introduced to prevent futile compression on small-context models) was checking `approx_tokens` — a message-only token estimate — instead of `approx_request_tokens` which includes tool schemas, system prompts, and other request overhead.

**What happened in practice:**
1. Agent session accumulates conversation history + tool definitions
2. Provider rejects with "Context size has been exceeded" because the full payload exceeds limits
3. Guard checks message-only tokens (~7,513) which appears below 64K floor
4. Compression is attempted despite being futile — same error repeats
5. After max attempts: "Context length exceeded and cannot compress further" — session dies

**Symptom:** Sessions crash with compression failure even when actual conversation content is small, because tool schemas inflate the real request size beyond what message-only estimates show.

#### The Fix

Both compression floor guards (413 payload-too-large handler at line 2799 and context-overflow handler at line 2894) now use `approx_request_tokens` instead of `approx_tokens`:

```python
# Before (broken):
if approx_tokens < MINIMUM_CONTEXT_LENGTH:
    # skip compression — spurious error

# After (fixed):
if approx_request_tokens < MINIMUM_CONTEXT_LENGTH:
    # skip compression — spurious error
```

This ensures the guard accounts for the full API payload including tool schemas, system prompts, and other overhead that contributes to actual request size.

#### Affected Code Paths

- `agent/conversation_loop.py` line 2799 (413 Payload Too Large handler)
- `agent/conversation_loop.py` line 2894 (Context Overflow handler)

## Full Disclosure

This bug was introduced during the upstream sync on June 14, 2026 (commit `15f37ac2e`) which added the `MINIMUM_CONTEXT_LENGTH` guard as a safety measure against premature compression. The guard logic itself was sound — it correctly prevents compression when tokens are below the floor. However, the wrong token estimate variable was used for the comparison.

The root cause: two separate token estimators exist in conversation_loop.py:
- `approx_tokens` (line 851) — estimates message content only via `estimate_messages_tokens_rough()`
- `approx_request_tokens` (lines 852-854) — full request estimate including tool schemas via `estimate_request_tokens_rough()`

The guard was written to check the cheaper message-only estimate, but the provider's "context exceeded" error is triggered by the full payload. When tool schemas add significant overhead (common with Lycus' extensive toolset), the discrepancy between these two estimates can be large enough to cause false negatives in the guard.

**Impact:** Users with large-context models (>128K) and heavy tool configurations were most affected, as the gap between message-only and full-request token counts is widest when many tools are registered. Sessions would crash unexpectedly during long-running tasks even though actual conversation content was well within limits.
